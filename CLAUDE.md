# haley-ai-darkroom 專案守則

## 專案簡介

一套 AI 輔助修圖／暗房系統，核心功能：

1. 依據已建立的人臉資料，自動辨識照片中的人物
2. 自動判定圖片中已建立的人臉與主體範圍
3. 根據分析結果，建議是否需要修圖，以及建議的裁切格式
4. 分析圖片參數（曝光、白平衡、對比等），提供自動調整建議
5. 依據成品評分結果，自動學習並優化調色風格
6. 建立完整的圖片資料庫（人臉、主體、參數、評分紀錄等）

資料庫：MySQL

## 程式碼撰寫規範

### 1. 檔案標頭：路徑註解

每個新建立的程式檔案，**第一行**必須以該語言的註解語法標註此檔案在專案中的相對路徑。範例：

```python
# src/face_recognition/detector.py
```

```javascript
// src/services/imageAnalysis.js
```

```sql
-- db/migrations/001_create_faces_table.sql
```

```html
<!-- templates/upload.html -->
```

修改既有檔案時，若該檔案尚未有此標頭，應一併補上；若路徑因搬移而改變，應同步更新。

### 2. 中文設計註解

每一段程式碼（函式、類別、關鍵邏輯區塊）都需要附上繁體中文註解，說明：

- **作用**：這段程式碼在做什麼
- **設計原因**：為什麼這樣寫、選擇這個做法的考量（例如效能、準確度、資料庫結構限制等）

避免逐行翻譯程式碼在做什麼（那是程式碼本身該表達的），重點放在「為什麼」這種不容易從程式碼本身看出來的資訊。

### 3. 參數抽離規範

- **機密／敏感參數**（資料庫帳密、API 金鑰、第三方服務密鑰等）一律放在 `.env`，並確保 `.env` 已列於 `.gitignore`，程式中以環境變數方式讀取，絕不硬編碼（hardcode）在原始碼中。
- **可調整的一般數值**（例如模型信心分數門檻、裁切比例預設值、批次大小、影像處理參數等）一律寫入 YAML 設定檔（例如 `config/settings.yaml`），不要寫死在程式碼裡，方便後續調整與實驗，不需改程式碼即可調整行為。
- 判斷原則：會外洩就會出事的 → `.env`；會想調整但不涉及機密的 → YAML。

### 4. 檔名唯一性

專案內所有檔案的**檔名**（不含路徑）必須保持唯一，不可有兩個不同目錄下卻同名的檔案（例如不可同時存在多個 `README.md`）。新增檔案前應先確認專案內是否已有同名檔案，若有需求類似但需放在不同目錄，應以更具體的名稱區分（例如 `IMAGE_DATA_GUIDE.md` 而非再取名 `README.md`）。

例外：`.gitkeep` 這類無內容、純粹用來讓 git 追蹤空目錄的標記檔，本來就會在多個目錄下重複出現，不受此規則限制。

### 5. 時區

所有程式中記錄的時間（分析時間戳記、資料庫時間欄位等）一律使用**台北時區（Asia/Taipei，UTC+8）**，不要用 UTC 或伺服器預設時區。Python 中請用 `zoneinfo.ZoneInfo("Asia/Taipei")`，MySQL 資料表若使用 `CURRENT_TIMESTAMP` 這類預設值，需確認資料庫連線／伺服器時區設定也對應到 +8，避免程式端與資料庫端時間對不上。

### 6. 其他一般規範

- 遵循 Claude Code 預設工程原則：不過度設計、不寫用不到的錯誤處理、不引入非必要的抽象層。
- 若在開發過程中發現有值得長期遵守、且尚未寫入本檔案的規則或決策，應主動提出建議寫入 CLAUDE.md。
